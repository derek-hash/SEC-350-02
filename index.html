// 03 — Security Controls

WAF + PKI + SIEM

WAF
ModSecurity
Web Application Firewall

ModSecurity deployed on nginx with SecRuleEngine On. Protects against SQLi, XSS, path traversal, and command injection. nginx has no outbound internet access — confirmed via firewall rules. HTTPS served via cert issued by derek-CA.

PKI
Enterprise CA
derek-CA Root Certificate

Enterprise Root CA on mgmt02. Signed certs distributed to nginx for HTTPS and jump for SSH CA auth. CA root imported on wks01 trusted root store. Passwordless SSH to jump uses TrustedUserCAKeys — certs signed with absolute date ranges to avoid clock skew issues.

All DMZ, LAN, and MGMT hosts forward logs to Wazuh at 172.16.200.10. Wazuh agents active on all Linux hosts and Windows VMs. MITRE ATT&CK mapping enabled — T1078 Valid Accounts detected with 1970+ hits in the dashboard.

// 06 — Troubleshooting

Issues & Exact Fixes

golden-rules // learned the hard way
RuleWhy It Matters
Absolute SSH cert datesUse 20200101000000:20300101000000 — never relative dates. VM clocks drift and break cert auth silently.
Mask systemd-timesyncdNTP resets jump's clock and invalidates certs every time. Stop, disable, and mask it permanently.
icacls Windows SSH keysWindows rejects any key accessible by SYSTEM or Administrators. Strip all ACLs and grant only the current user.
Apply VyOS zone policiesA ruleset with no zone policy applied does absolutely nothing. Always set firewall zone X from Y firewall name Z.
Check both firewallsTraffic crossing multiple zones touches both edge01 AND fw-mgmt. A rule on one means nothing without the other.
Ruleset names must match exactlyLAN-DMZ and LAN-to-DMZ are different names and will silently fail. VyOS won't warn you.
ERR
wazuh agents (nginx/jump) couldn't reach manager

Zone Policy Not Applied — DMZ-to-LAN

Ruleset existed but was never applied to the LAN zone's from DMZ direction. Also missing: static route on edge01 for the MGMT subnet.

edge01
ERR
SCP from jump → mgmt02 hanging

Missing Destination IP + OpenSSH Not Installed

LAN-to-MGMT rule 30 had no destination IP set. Separately, OpenSSH Server was not installed on mgmt02 — Windows doesn't ship it by default.

fw-mgmt
ERR
certificate invalid: not yet valid

NTP Resetting jump's Clock Below Cert Validity

systemd-timesyncd was resetting jump's clock to a time before cert validity start. Fix: mask NTP permanently AND re-sign certs with absolute wide date range.

jump
ERR
signature algorithm not supported

Ubuntu 24.04 Rejects RSA + ED25519 Certs by Default

OpenSSH on Ubuntu 24.04 doesn't accept these algorithms out of the box. Must explicitly add PubkeyAcceptedAlgorithms and CASignatureAlgorithms to sshd_config.

jump
ERR
Windows client ignoring cert / UsePAM overriding PasswordAuthentication no

Windows ACLs + PAM Bypass

Two separate issues: Windows SSH rejects keys if SYSTEM/Administrators have access — fix with icacls /inheritance:r. PAM also bypasses PasswordAuthentication no — must set UsePAM no.

traveler
ERR
mgmt01 → jump SSH hanging

Ruleset Name Mismatch — LAN-DMZ vs LAN-to-DMZ

Zone policy for DMZ had from LAN pointing to LAN-DMZ instead of LAN-to-DMZ. VyOS accepted it silently and dropped all traffic. Deleted the wrong entry and re-applied the correct name.

edge01
ERR
wks01 can't reach nginx / DNS timing out

Missing HTTP/S Rule + Missing DNS Rule

Two separate missing rules: no HTTP/HTTPS allow in LAN-to-DMZ on edge01, and no DNS allow in LAN-to-MGMT on fw-mgmt. Both had to be added and committed before wks01 could resolve and browse to nginx.derek.local.

wks01
ERR
certreq submit failing — RPC server unavailable

Wrong CA Config String

Used the wrong config string for the CA. Fix: run certutil -dump | findstr "Config" on mgmt02 to get the exact string, then pass it to certreq -submit -config. Correct value: mgmt02-derek.derek.local\derek-CA.
